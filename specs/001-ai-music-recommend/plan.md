# Implementation Plan: AI Music Recommendation via Vocal Analysis

**Branch**: `001-ai-music-recommend` | **Date**: 2026-03-29 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-ai-music-recommend/spec.md`

## Summary

Build a web-based AI music recommendation product where users record singing
or upload audio, the system extracts vocal features client-side (Meyda.js +
pitchfinder), sends feature vectors to a FastAPI backend for ML analysis
(librosa + scikit-learn), and generates explainable song direction
recommendations via a hybrid rule engine + LLM (Claude Sonnet) architecture.
No raw audio is stored on servers; only derived features persist.

## Technical Context

**Language/Version**: TypeScript 5.x (frontend), Python 3.11+ (backend)
**Primary Dependencies**:
- Frontend: Vite 6, React 19, React Router v7, Meyda.js, pitchfinder
- Backend: FastAPI, Uvicorn, Pydantic, librosa, scikit-learn, aiosqlite, Anthropic SDK
**Storage**: SQLite (WAL mode) via aiosqlite
**Testing**: Vitest (frontend), pytest (backend)
**Target Platform**: Modern browsers (Chrome 49+, Firefox 25+, Safari 14.1+, Edge 79+)
**Project Type**: Web application (frontend/backend separation)
**Performance Goals**: LCP < 2.5s, API p95 < 200ms (feature analysis), full pipeline < 30s
**Constraints**: Minimal libraries, no raw audio on server, explainable recommendations
**Scale/Scope**: 500 concurrent analysis sessions, 5 pages, ~30-40 song direction entries

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### I. Code Quality & Maintainability

- [x] **Readability first**: Component and module naming follows domain language
  (VocalProfile, AudioRecorder, RecommendationEngine).
- [x] **Detailed comments**: All services and analysis modules will have
  JSDoc/docstring documentation explaining purpose, parameters, returns.
- [x] **Consistent formatting**: ESLint + Prettier (frontend), Ruff (backend)
  configured in project setup.
- [x] **Library-first**: Meyda.js for audio features, librosa for server-side
  analysis, scikit-learn for classification. No reinventing FFT or MFCC.
- [x] **No dead code**: Enforced by linter rules (no-unused-vars, F841).
- [x] **Type safety**: TypeScript strict mode (frontend), Pydantic models +
  type annotations (backend).

### II. Testing Standards

- [x] **Unit tests**: Vitest for frontend services/utils, pytest for backend
  services/models.
- [x] **Integration tests**: pytest with FastAPI TestClient for API endpoints
  against real SQLite database.
- [x] **Contract tests**: API contract tests verifying request/response schemas
  match the OpenAPI spec auto-generated by FastAPI.
- [x] **Coverage threshold**: Configured in CI (minimum 80% for new code).
- [x] **Test independence**: Each test uses its own database fixture/context.

### III. User Experience Consistency

- [x] **Design system**: Shared CSS tokens (colors, spacing, typography)
  defined in a central design tokens file. All components use tokens.
- [x] **Responsive**: Mobile-first design, tested at 320px, 768px, 1024px.
- [x] **Accessibility**: WCAG 2.1 AA. Audio recorder has keyboard controls.
  Profile charts have text alternatives. Sufficient color contrast.
- [x] **Loading states**: Recording shows waveform, analysis shows progress
  indicator, LLM generation shows skeleton cards.
- [x] **Error communication**: User-facing error messages in plain language.
  Technical details logged server-side only.
- [x] **Consistent patterns**: Navigation, recording flow, result display
  follow the same interaction model across all pages.

### IV. Performance Requirements

- [x] **Page load**: Vite code-splitting + lazy loading. Meyda.js loaded
  only when recording starts. Target LCP < 2.5s.
- [x] **API response**: Feature analysis endpoint < 200ms p95 (receives
  pre-extracted feature vectors, not raw audio). LLM call adds 2-5s but
  runs asynchronously with rule engine results shown first.
- [x] **Bundle size**: Meyda (~15KB) + pitchfinder (~5KB) + React Router.
  No heavy dependencies. Tree-shaking enabled.
- [x] **Database queries**: Simple queries (insert session, fetch history).
  All queries use indexes on user_id and created_at.
- [x] **Monitoring**: FastAPI middleware logs response times. Health check
  endpoint at /api/v1/health.

### Architecture Constraints

- [x] **Decoupled deployments**: Frontend (Vite build → static files) and
  backend (FastAPI + Uvicorn) are independently deployable.
- [x] **API-first**: All interaction via REST API at /api/v1/*.
- [x] **Shared contracts**: FastAPI auto-generates OpenAPI schema. Frontend
  TypeScript types derived from the schema.
- [x] **Independent stacks**: TypeScript/React frontend, Python/FastAPI backend.
- [x] **CORS and security**: FastAPI CORS middleware configured. JWT tokens
  for authentication (P3 feature).
